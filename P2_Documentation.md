## 1. Overview
The **UserOps** module implements the "Time Layer" of the Lattice-Based Forward Secure IBE project. It is responsible for managing the lifecycle of user keys (Forward Security) and performing the actual data encryption/decryption using Lattice primitives.

## 2. Class: `UserOps`
This class encapsulates all user-side operations. It is initialized with the `system_setup` dictionary provided by the **P1 (LatticeCrypto)** module, which contains:
* `params`: Lattice parameters ($n, q, \sigma$).
* `A`: Public Matrix.
* `T_A`: Master Trapdoor (used for key simulation).
* `tree`: The Binary Tree structure defining the time epochs.

## 3. Forward Security Logic

### 3.1. Minimal Cover Mechanism (`get_min_cover`)
**Objective:** Determine the precise set of keys a user must hold to access data from `current_time` onwards, without having access to the past.

**Algorithm:**
The function performs a recursive traversal of the binary tree:
1.  **Input:** `current_time` $t$.
2.  **Target Range:** $[t, \text{TotalEpochs} - 1]$.
3.  **Recursive Check:** For each node:
    * If the node is completely in the **past**, it is ignored.
    * If the node is completely in the **future** (or current range), it is added to the cover (this node acts as a master key for its entire subtree).
    * If the `current_time` **splits** the node, the function recurses into the Left and Right children.

### 3.2. Key Update (`Update`)
**Objective:** Evolve the user's key bundle from time $t$ to $t+1$.

**Workflow:**
1.  Calculate `needed_nodes` for $t+1$ using `get_min_cover`.
2.  Initialize `new_key_bundle`.
3.  **Key Retention:** If a key exists in the old bundle and is required for the new one, it is preserved.
4.  **Key Deletion:** Keys in the old bundle that are *not* in `needed_nodes` are discarded. This is the cryptographic deletion step that ensures Forward Secrecy.
5.  **Key Derivation:** In this simulation, missing keys are flagged to be generated by the helper function.

## 4. Cryptographic Operations

### 4.1. Dual Regev Encryption (`Encrypt`)
Implements the Learning With Errors (LWE) based encryption scheme.

**Steps:**
1.  **Targeting:** The recipient's ID and Time Epoch are hashed to a lattice vector $u$:
    $$u = H(\text{UserID} || \text{Epoch})$$
2.  **Noise Generation:**
    * Secret vector $s \in \mathbb{Z}_q^n$.
    * Error vector $e_1$ (size $m$) and scalar $e_2$ sampled from a Discrete Gaussian distribution.
3.  **Ciphertext Construction:**
    * $c_1 = A^T s + e_1$
    * $c_2 = u^T s + e_2 + \text{MessageEncoding}$
    * Message Encoding: Bit 0 maps to 0, Bit 1 maps to $\lfloor q/2 \rfloor$.

### 4.2. Decryption (`Decrypt`)
Recovers the message using the secret key $sk$ corresponding to the ciphertext's epoch.

**Steps:**
1.  **Key Lookup:** Finds the valid secret key $sk$ in the user's bundle for the target epoch.
2.  **Regev Decryption:**
    $$M_{approx} = c_2 - \langle sk, c_1 \rangle$$
    This works because $A \cdot sk = u$, canceling out the main term and leaving only the noise + message.
3.  **Thresholding:** The noisy message $M_{approx}$ is mapped to the nearest bit (0 or 1) based on its distance to 0 or $q/2$.

## 5. Simulation Helper
### `simulate_key_evolution`
In a fully deployed HIBE (Hierarchical IBE), child keys are derived mathematically from parent keys. Since the base P1 module provides "Root Extraction" (`SamplePre`) but not "Basis Delegation", this function maintains the simulation's integrity by:
1.  Using the Master Trapdoor to generate valid keys for the specific nodes identified by the Update logic.
2.  This allows the system to demonstrate the correct **Forward Secure Workflow** (Cover sets, Updates, Deletion) without requiring the full complexity of Lattice Delegation algorithms.
